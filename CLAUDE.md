## System Overview

BBGO SaaS is a multi-tenant cryptocurrency trading platform. Five components work together:

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────────────────────────┐
│  Web (Next)  │────▶│  Manager (Go)    │────▶│  BBGO Instances (per-strategy)      │
│  :3142       │     │  :8090 (Docker)  │     │  Each strategy = own Docker container│
└─────────────┘     │                  │     │  live + paper (Docker)               │
                    │  ┌────────────┐  │     └─────────────────────────────────────┘
                    │  │ MarketData │◀─┼──── BBGO MarketData (shared, Docker)
                    │  │   Hub ×2   │  │     mainnet :9090 + testnet :9090
                    │  └────────────┘  │
                    └────────┬─────────┘
                             │
                    ┌────────▼─────────┐
                    │  Supabase (DB)   │
                    │  + Realtime       │
                    └──────────────────┘
```

| Component | Tech | Runs | Role |
|-----------|------|------|------|
| **Web** | Next.js 16 + TypeScript | `pnpm dev` (local) | User-facing dashboard, strategy config, charts |
| **Manager** | Go + Chi | Docker container | Instance orchestration, credential encryption, data sync, backtest execution, WebSocket relay |
| **BBGO Instances** | Go (bbgo binary) | Docker (per-strategy) | Trading engine — one container per strategy instance |
| **Market Data** | Go (bbgo binary, gRPC) | Docker (2 instances) | Shared kline/orderbook data via gRPC; mainnet + testnet |
| **Supabase** | PostgreSQL + Realtime | Cloud | Persistent storage + realtime subscriptions for live/paper data |

### Core Business Flows

1. **Paper/Live Trading** — Users create strategies, manager spawns one Docker container per strategy instance, bbgo writes data directly to Supabase
2. **Backtests** — Users submit backtest configs, manager runs isolated `docker run --rm` backtest jobs, results stored in Supabase
3. **Market Data** — Shared service provides real-time market data to all containers and WebSocket clients
4. **Realtime Updates** — Supabase Realtime broadcasts new trades/orders/positions/profits to the frontend

## Business Flow 1: Paper/Live Trading

### Per-Instance Architecture

Each strategy gets its own **Docker container** (not a shared user container). Container names follow `bbgo-{userID}-{instanceID}`. The `?mode=live|paper` query parameter selects which mode's data to query (defaults to `live`).

Strategy instances are tracked in the `strategy_instances` Supabase table with:
- Deterministic `instance_id` computed from strategy + symbol + config via `pkg/instanceid`
- Per-instance `strategy_instance_id` propagated to orders and trades for data isolation
- Live and paper instances stored separately (partitioned by `mode` column)

### Request Flow

```
Browser → Next.js (/api/manager/*)
  → Supabase auth check
  → Forward to Manager (adds X-User-Id + X-Manager-Token headers)
    → Manager validates token, resolves user
    → Bot data: queries Supabase directly or proxies to running bbgo container
    → Strategies: manages Docker containers per instance
    → Real-time: WebSocket + Supabase Realtime subscriptions
```

### Instance Lifecycle

```
stopped → starting (HTTP 202) → running → stopped
                            ↘ error
```

`POST /api/users/{userID}/instances/{instanceID}/start` returns 202 immediately. Background goroutine handles container creation + health check polling + gRPC readiness.

Container recovery (`container_recovery.go`) periodically checks running containers and restarts any that have died, using a goroutine pool (max 5 parallel).

### Data Flow

```
Exchange → BBGO Container → Supabase (orders, trades, positions, profits)
                            ↘ Paper mode: writes to paper_* tables with SUPABASE_TABLE_PREFIX=paper_
```

BBGO writes directly to Supabase when `DB_DRIVER=supabase` is set. Paper containers write to `paper_orders`, `paper_trades`, `paper_positions`, `paper_profits` tables via `SUPABASE_TABLE_PREFIX=paper_`. Frontend reads from the appropriate table based on the current trading mode.

Paper trading now supports **futures and margin** modes in addition to spot. When a strategy session has `Futures=true` or `Margin=true`, the paper trade engine activates futures position tracking (short selling, leverage-based margin locking, unrealized PnL, liquidation price computation) or margin simulation (borrow/repay assets, hourly interest accrual). Futures position risks are synced to `paper_futures_position_risks` every 30s via background goroutine. Margin interest accrues hourly at 0.01% rate. See `pkg/bbgo/paper_trade_futures.go` for implementation details.

Supabase Realtime broadcasts changes to the frontend for live updates (migration `00023_realtime_tables.sql`).

### Frontend Trading Mode

Global **live/paper toggle** in header via `TradingModeProvider` context. `useTradingMode()` hook returns `{ mode, setMode }` — used across all data queries. Persisted to `localStorage` with cross-tab sync.

Data queries use `tableName(base, mode)` from `supabase-adapters.ts` which prefixes table names with `paper_` for paper mode.

### Credential Management

- API keys encrypted with AES-GCM (`crypto.go`), stored per-user on disk (`{DATA_DIR}/{userID}/credentials.json`)
- Credential verification via HMAC-SHA256 signed requests for Binance/Bybit/Bitget/OKEx/KuCoin (`verify.go`)
- `envArgs()` in `container.go` injects mode-specific credentials into containers
- Live containers never get `PAPER_TRADE` or `BINANCE_TESTNET` env vars (verified in tests)

### Strategy System (50 strategies)

Strategy definitions come from three sources:
- **Strategy Registry** (`strategy_registry` Supabase table) — centralized defaults, field definitions, liveOnly flags, requiresFutures flags. Cached in manager via `StrategyDefaultsCache` (`strategy_registry.go`).
- **Frontend**: `web/src/lib/bbgo/strategies.ts` — `STRATEGY_SCHEMAS` array with UI metadata, categories, nested field definitions
- **BBGO**: `pkg/strategy/<name>/strategy.go` — `bbgo.RegisterStrategy(ID, &Strategy{})`

**Strategy type generation**: `web/scripts/gen_strategy_types.mjs` reads `strategies.ts` and generates `manager/strategy_types.go` with typed Go structs. Regenerate with `pnpm gen-strategy-types`.

**liveOnly (22 strategies):** Frontend shows warning + auto-switches to live mode; Manager blocks paper creation with HTTP 400. Both lists must stay in sync.

**Cross-exchange (10 strategies):** Frontend collects `sessionRoles`; Manager creates sessions per role. Same API key for spot+futures on same exchange.

**Legacy aliases:** `user.go:legacyStrategyAliases` and `legacyFieldAliases` normalize old strategy IDs/field names.

**Nested fields**: Frontend strategies use flat field keys with dot notation (e.g., `profitSpread`). `strategy_registry.go` applies deep-merge defaults and the config form maps flat → nested before sending to manager.

## Business Flow 2: Backtests

### Architecture

Backtests run as **isolated Docker containers** (`docker run --rm`) on the manager host. Each job gets its own config directory (`/data/backtest-{userID}/{jobID}/`) and database file to prevent concurrent overwrite. Containers are named `bt-{jobID}` and auto-removed after completion.

### Job Lifecycle

```
pending → downloading (market data) → running → completed
                                                ↘ failed
```

Semaphore limits to 2 concurrent backtests. Stale jobs from previous manager restarts are auto-failed on startup.

### Flow

1. User submits backtest config (strategy, exchange, symbol, date range)
2. Manager downloads market data from shared market data service (gRPC)
3. Manager runs `docker run --rm` with isolated config/DB (SQLite, not Supabase)
4. Results parsed from output directory and stored in `backtest_reports` table
5. Equity curve and CSV uploaded to Supabase Storage for download
6. User can view results (equity chart, P&L, trades) in the frontend

### Strategy Defaults

Before running, backtest configs are enriched with defaults from the strategy registry (`StrategyDefaultsCache`). Missing fields like `symbol`, `quantity`, `gridNumber`, `interval` etc. are injected to prevent strategy panics.

### Key Files

| File | Purpose |
|------|---------|
| `manager/backtest_job.go` | `BacktestJobStore` (job queue, persistence) + `BacktestExecutor` (run, sync, upload) |
| `manager/container.go` | `ContainerManager.RunBacktest()` — Docker run --rm with isolated config/DB |
| `web/src/components/backtest/BacktestEquityChart.tsx` | Equity curve chart |
| `web/src/components/user/BacktestPanel.tsx` | Backtest form and results |

## Business Flow 3: Market Data

### Shared Market Data Service

Two long-running bbgo instances provide market data to all users:
- **Mainnet** (`bbgo-marketdata:9090`) — production exchange data
- **Testnet** (`bbgo-marketdata-testnet:9090`) — testnet exchange data

Both run with `--enable-grpc` and expose kline/orderbook/ticker data via gRPC. The 3-layer kline cache (memory → SQLite → API) is used with `KLINE_DB_PATH` for disk persistence.

### Manager MarketDataHub

Two `MarketDataHub` instances (`marketdata.go`) — one for mainnet, one for testnet:
- `hubForMode(mode)` routes to the correct hub based on container mode
- Broadcasts market data to WebSocket clients
- Provides market data to backtest jobs for historical data download
- Pooled gRPC connections per user container with reference counting
- Default subscriptions configured via `MARKET_SUBSCRIPTIONS` env var

## Database

### Supabase Schema (`supabase/migrations/`, 23 migrations)

Core tables:

| Table | Written By | Purpose |
|-------|-----------|---------|
| `user_profiles` | Web (auth trigger) | User metadata |
| `exchange_credentials` | Manager | Encrypted API keys |
| `strategy_registry` | Migration + manual | Strategy definitions, defaults, fields (single source of truth) |
| `strategy_instances` | Manager | Per-instance container tracking (replaces file-based store) |
| `orders` | BBGO (direct) | Live order records |
| `trades` | BBGO (direct) | Live trade records |
| `positions` | BBGO (direct) | Live position snapshots |
| `profits` | BBGO (direct) | Live profit snapshots |
| `paper_orders` | BBGO (direct) | Paper order records (mirror of orders) |
| `paper_trades` | BBGO (direct) | Paper trade records (mirror of trades) |
| `paper_positions` | BBGO (direct) | Paper position snapshots (mirror of positions) |
| `paper_profits` | BBGO (direct) | Paper profit snapshots (mirror of profits) |
| `nav_history_details` | BBGO (direct) | Account NAV history |
| `rewards` / `withdraws` / `deposits` | BBGO (direct) | Account activity |
| `margin_loans` / `margin_repays` / `margin_interests` / `margin_liquidations` | BBGO (direct) | Margin trading records |
| `futures_position_risks` | BBGO (direct) | Futures position risk snapshots |
| `paper_futures_position_risks` | BBGO (direct, paper mode) | Paper futures position risk snapshots (synced every 30s) |
| `paper_margin_loans` / `paper_margin_repays` / `paper_margin_interests` / `paper_margin_liquidations` | BBGO (direct, paper mode) | Paper margin trading records |
| `backtest_reports` | Manager | Backtest results (JSONB) |
| `user_containers` | Manager | Legacy container state |

Paper tables (`paper_*`) mirror live tables exactly with the same schema, RLS policies, and indexes. BBGO writes to paper tables when `SUPABASE_TABLE_PREFIX=paper_` is set.

Service parity tables (`00022_service_parity_tables.sql`) create `paper_` mirrors for all account-service tables.

Realtime publication (`00023_realtime_tables.sql`) enables Supabase Realtime for all live and paper trading tables.

- Orders and trades include `strategy_instance_id` column for per-instance data isolation
- Orders use `order_type` column (not `type`) — matches bbgo's SQLite schema
- Multi-tenant via `user_id` column on all tables
- RLS policies restrict to owning users; service_role for cross-user operations

### Docker Networking

All backend services share `saas_bbgo-net` bridge network:
- Manager → BBGO instances: `http://bbgo-{userID}-{instanceID}:{BBGO_PORT}`
- Manager → Market Data: `bbgo-marketdata:9090` (gRPC), `bbgo-marketdata:8080` (REST)
- Shared volume `bbgo-data` at `/data` for configs, databases, credentials, backtest data
